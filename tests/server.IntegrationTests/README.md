# Документація інтеграційних тестів (MVP)

Цей файл містить детальне пояснення розроблених інтеграційних тестів для базового функціоналу застосунку. Ми використовуємо підхід **User Journeys** (сценаріїв користувача), щоб тестувати фічі у реалістичній послідовності.

## 1. Інфраструктура тестування

### `IntegrationTestCollection.cs`
> [!IMPORTANT]
> xUnit за замовчуванням запускає всі тестові класи паралельно. Оскільки кожен клас використовує свій екземпляр `WebApplicationFactory`, це призводило до конфліктів при міграції бази даних (дві бази намагалися ініціалізуватися одночасно і перезаписували глобальні змінні середовища). 
Щоб вирішити цю проблему, ми створили `CollectionFixture`. Тепер усі інтеграційні тести виконуються послідовно і **використовують одну й ту саму** тимчасову БД у Docker, що значно прискорює їх виконання.

### `IntegrationTestBase.cs`
Базовий клас для всіх тестів. Він містить допоміжні методи:
- `SendGraphQLAsync`: Надсилає HTTP POST запити до GraphQL. Підтримує передачу `Authorization: Bearer` токена.
- `RegisterAndLoginAsync`: Реєструє користувача і автоматично підтверджує його пошту через прямий SQL-запит до `Testcontainers` бази (через Dapper). Це дозволяє тестам миттєво авторизовуватися без необхідності піднімати реальний SMTP сервер.
- `CreateHelpRequestAsync`: Допоміжний метод для швидкого створення заявки, оскільки це потрібно майже в кожному тесті.

> [!TIP]
> У `IntegrationTestBase` ми вимкнули обробку Cookies (`HandleCookies = false`) в `HttpClient`. Якби ми цього не зробили, сервер міг би сплутати різних користувачів (наприклад, волонтера з творцем заявки), оскільки `HttpClient` запам'ятовував би JWT токен першого користувача і відсилав його в куках для всіх наступних.

## 2. Життєвий цикл заявки (`HelpRequestLifecycleTests.cs`)

Цей клас тестує два основних потоки:

### `HappyPath_Create_Respond_Assign_Complete_Report_Delete_ShouldWork`
Перевіряє найідеальніший сценарій (Happy Path), коли заявка проходить всі етапи без перешкод.
1. **Реєстрація:** Створюються два користувачі: Власник та Волонтер.
2. **Створення:** Власник публікує заявку.
3. **Відгук:** Волонтер знаходить заявку та залишає відгук (`respondToHelpRequest`).
4. **Перегляд кандидатів:** Власник робить запит на отримання списку кандидатів (`helpRequestResponses`) і бачить Волонтера.
5. **Затвердження:** Власник обирає Волонтера (`assignExecutor`).
6. **Завершення:** Статус заявки змінюється на `RESOLVED`.
7. **Звіт:** Власник залишає звіт про допомогу (`createReport`).
*(Примітка: SignalR етапи будуть тестуватися в окремому класі)*

### `OpenRequest_Edit_Cancel_Restore_SoftDelete_ShouldWork`
Перевіряє можливості маніпуляції з заявкою, поки вона ще знаходиться у статусі `Open` і не має призначеного виконавця.
1. **Редагування:** Зміна назви та опису.
2. **Скасування:** Переведення в статус `Cancelled`.
3. **Відновлення:** Повернення заявки в `Open`.
4. **Soft Delete:** Успішне видалення відкритої заявки.

## 3. Управління кандидатами (`HelpRequestAssigneeManagementTests.cs`)

Цей клас тестує взаємодії між власником заявки та потенційними або вже призначеними виконавцями:
- **Скасування відгуку кандидатом**: Волонтер відгукується, а потім вирішує скасувати свій відгук. Перевіряється, що статус відгуку успішно змінюється на `CANCELLED`.
- **Відмова призначеного виконавця**: Волонтера вже призначено виконавцем заявки (заявка в статусі `InProgress`), але він відмовляється з певної причини.
- **Зняття виконавця власником**: Власник заявки вирішує самостійно зняти призначеного волонтера і вказує причину.

## 4. Обмеження профілю користувача (`UserProfileTests.cs`)

Тут перевіряються обмеження на видалення акаунту (Soft Delete):
- **Заборона видалення при активних заявках**: Користувач не може видалити свій профіль (сервер повертає `User.HAS_ACTIVE_REQUESTS_AS_OWNER`), якщо він є власником хоча б однієї відкритої заявки, або є призначеним виконавцем іншої заявки. Під час розробки ми також виявили та виправили баг з перетворенням типу при перевірці адміністратора в базі даних.
- **Успішне видалення**: Якщо у користувача немає активних заявок, профіль успішно "м'яко" видаляється.

## 5. Чат та етапи виконання (`ChatAndStagesTests.cs`)

Найскладніший тест, який перевіряє інтеграцію **SignalR** в нашому `WebApplicationFactory` та взаємодію між користувачами в режимі реального часу.
- Підключення двох користувачів (`creator` та `volunteer`) до `ChatHub` використовуючи їхні JWT токени.
- Надсилання звичайних повідомлень у чат (`SendMessage`) та перевірка отримання через подію `ReceiveMessage`.
- Пропозиція етапу (`ProposeStage`) власником.
- Успішне підтвердження етапу (`ConfirmStage`) волонтером та перевірка виклику відповідної події.
Цей тест підтверджує, що WebSocket-з'єднання та інфраструктура SignalR працюють коректно.

> [!NOTE]
> Всі 9 інтеграційних тестів на даний момент успішно проходять (Exit Code 0). Усі заплановані сценарії MVP покриті тестами.
